## 1. Módulo REP — Obligaciones normativas en el modal de productor

**Problema**
El modal de agregar/editar productor no tenía soporte para seleccionar las obligaciones normativas aplicables a cada productor.

**Solución implementada**
Se creó el modelo `NormativeObligation` en el backend y se añadió una sección de checkboxes en el modal del frontend. Las obligaciones se almacenan como array de strings en `obligaciones_normativas` del productor.

**Seeds de obligaciones** (ejecutados en `initial_data.py` en cada deploy):
- "Envases y empaques (Res. 1407)"

## 2. Módulo REP — Departamentos y municipios de Colombia (datos estáticos)

**Problema**
El modal de productor no tenía selector geográfico para departamento y ciudad.

**Solución implementada**
Se creó un archivo de datos estáticos con los 33 departamentos y sus municipios según el DANE. El selector de ciudad carga dinámicamente según el departamento seleccionado.

**Archivos modificados**

| Archivo | Cambio |
|---|---|
| `frontend/src/app/core/data/colombia-geo.ts` | `COLOMBIA_GEO`, `COLOMBIA_DEPTOS`, `getCitiesForDepto()` |
| `frontend/src/app/features/dashboard/producers/producers.ts` | `departamentoValue = toSignal(...)`, `availableCities = computed(...)`, effect de reset de ciudad |
| `frontend/src/app/features/dashboard/producers/producers.html` | `<select>` de departamento + `<select>` de ciudad condicionado a `availableCities()` |

**Decisión**: datos en el frontend (no en DB) porque la geografía colombiana no cambia y no requiere administración. Archivo estático evita un endpoint y una tabla extra.

**Comportamiento**: al cambiar de departamento, si la ciudad actual no pertenece al nuevo departamento, el campo ciudad se limpia automáticamente.

## 3. Módulo REP — Validación de campos obligatorios en el modal

**Problema**
El modal de productor no mostraba errores al enviar con campos vacíos.

**Solución implementada**
Se añadió `Validators.required` a todos los campos del formulario y se aplicó el patrón Bootstrap `is-invalid` / `invalid-feedback` en el template.

**Campos con `Validators.required`**: `sector_id`, `razon_social` (+ min/max length), `nit` (+ min/max length), `ciudad`, `departamento`, `direccion`, `correo` (+ `Validators.email`), `contacto`, `nombre_responsable`, `tipo`, `cuenta_con_plan`.

**Campo sin required**: `en_incumplimiento_rep` (boolean con default `false`).

**Comportamiento**: `submitProducer()` llama `form.markAllAsTouched()` si el formulario es inválido, lo que dispara la visualización de todos los errores antes de regresar.

## 4. Módulo REP — Separación Agregar/Editar + múltiples productores por usuario

**Problema**
El botón "Agregar" funcionaba como upsert (creaba o editaba el productor propio). No había forma de editar un productor específico de la tabla ni lógica para múltiples productores.

**Solución implementada**

### Backend
- Eliminado `UniqueConstraint("owner_id")` de `Producer` → permite N productores por usuario.
- Nuevas funciones CRUD: `get_producers_by_owner`, `create_producer`, `update_producer_by_id`.
- Nuevos endpoints:
  - `GET /rep/producers` → productores del usuario actual (`owner_id = current_user.id`)
  - `POST /rep/producers` → crear productor (solo `demo_col` o superuser)
  - `PUT /rep/producers/{producer_id}` → editar productor propio (o superuser)
- `demo_col` añadido a `_REP_ALLOWED_PLANS` (antes solo `demo_rep`).

### Frontend
- `openModal(content, mode: 'add' | 'edit', row?)` — dos modos:
  - `'add'`: resetea el form, `editingProducer.set(null)`
  - `'edit'`: parchea el form con los datos de `row`, `editingProducer.set(row)`
- `canAddProducer` computed:
  - `isSuperAdmin()` → siempre puede
  - `planType() === 'demo_col'` → siempre puede
  - Cualquier otro plan → solo si `myProducers().length === 0`
- `submitProducer()` despacha:
  - `updateProducerById` si hay `editingProducer()`
  - `addProducer` si es admin o `demo_col` (nuevo productor)
  - `saveMyProducer` para `demo_rep`/`demo_indv` (upsert del propio)
- Tabla: columna de acciones con botón editar (lápiz) por fila.
- Título del modal dinámico: "Agregar productor" / "Editar productor".

**Archivos modificados**

| `backend/app/crud/producer.py` | `get_producers_by_owner`, `create_producer`, `update_producer_by_id` |
| `backend/app/api/routes/rep.py` | `_REP_ALLOWED_PLANS` incluye `demo_col`; nuevos endpoints GET/POST/PUT |

## 5. Corrección — Cada usuario ve solo sus propios productores

**Problema**
Se había implementado un endpoint `GET /rep/producers/all` y lógica en el frontend para que el superadmin viera todos los productores de todos los usuarios. El requerimiento real es que cada usuario (incluyendo el superadmin) solo vea sus propios productores.

**Solución implementada**
Eliminada toda la lógica de `allProducers` del componente. La tabla siempre muestra `myProducers` (filtrados por `owner_id = current_user.id` en el backend).

**Archivos modificados**

| Archivo | Cambio |
|---|---|
| `frontend/src/app/features/dashboard/producers/producers.ts` | `tableRows`/`tableLoading` siempre usan `myProducers`; eliminado effect `loadAllProducers`; `refreshTable()` simplificado |

| `frontend/src/app/store/rep/rep.reducer.ts` | `addProducerSuccess` y `updateProducerByIdSuccess` ya no actualizan `allProducers` |

## 6. Corrección — NG8107 en template del componente Producers

**Problema**
El build generaba 17 advertencias `NG8107` ("left side of optional chain does not include null/undefined") en todas las expresiones `producer()?.` del template.

**Causa raíz**
TypeScript, sin `noUncheckedIndexedAccess` en `tsconfig`, infiere `arr[0]` como `T` (no `T | undefined`). Por eso `computed(() => this.myProducers()[0] ?? null)` era inferido como `Signal<ProducerRead>` en lugar de `Signal<ProducerRead | null>` — la cadena `?? null` se vuelve no-op para el sistema de tipos. El template checker de Angular veía `producer()` como `ProducerRead`, haciendo redundantes todos los `?.`.

**Solución**
Añadir tipo de retorno explícito al computed:

```typescript
readonly producer = computed((): ProducerRead | null => this.myProducers()[0] ?? null);
```

**Archivo modificado**: `frontend/src/app/features/dashboard/producers/producers.ts`
