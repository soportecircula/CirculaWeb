## 3. Seguridad del Login

### 3.1 Checkbox "Recordarme"

#### Decisión de arquitectura

La persistencia de sesión **no** cambia el storage del access token (sigue en memoria). Lo que cambia es el `max_age` del refresh token cookie:

| Estado del checkbox | Cookie | Comportamiento |
|---|---|---|
| Desmarcado | Session cookie (`max_age=None`) | Se borra al cerrar el navegador |
| Marcado | Persistent cookie (`max_age=7 días`) | Sobrevive al cierre del navegador |

El email del usuario se guarda en `localStorage` bajo la clave `circula_remember_email` únicamente para pre-rellenar el campo en la próxima visita. **Nunca se almacena el token ni información sensible en localStorage.**

#### Archivos modificados — Backend

**`backend/app/api/routes/auth.py`**

`_set_refresh_cookie` acepta ahora el parámetro `persistent: bool = True`:

```python
def _set_refresh_cookie(response, refresh_token, persistent=True):
    response.set_cookie(
        max_age=REFRESH_TOKEN_EXPIRE_MINUTES * 60 if persistent else None,
        ...
    )
```

**`backend/app/api/routes/login.py`**

Añadido `remember_me: bool = Query(False)` al endpoint `POST /login/access-token`. Se pasa a `_set_refresh_cookie(persistent=remember_me)`.

#### Archivos modificados — Frontend

> **Restricción:** `frontend/src/client/` es generado por ng-openapi y no se edita manualmente. El `LoginService` generado no soporta el query param `remember_me`, por lo que el effect de login fue reescrito para usar `HttpClient` directamente.

**`frontend/src/app/store/Authentication/authentication.actions.ts`**

### 3.2 Bloqueo por Intentos Fallidos

#### Decisión de arquitectura

El bloqueo es **por email** (no por IP) y vive en **Redis**. Se eligió un cliente Redis **síncrono** (`redis.Redis`) para mantener el endpoint de login como función síncrona FastAPI sin necesidad de `asyncio`.

**Comportamiento:**

| Evento | Acción |
|---|---|
| Credenciales incorrectas (intento 1–4) | Incrementa contador en Redis; respuesta incluye intentos restantes |
| Credenciales incorrectas (intento 5) | Bloquea la cuenta; responde 429 con mensaje de bloqueo |
| Intento mientras bloqueado | Responde 429 inmediatamente sin verificar credenciales |
| Login exitoso | Resetea el contador y elimina el bloqueo |

**Claves Redis:**

```
circula:{env}:login_fails:{email_lowercase}  → conteo (int), TTL = LOGIN_BLOCK_MINUTES
circula:{env}:login_block:{email_lowercase}  → "1", TTL = LOGIN_BLOCK_MINUTES
```

**Fail-open:** si Redis no responde, `is_blocked()` devuelve `False` e `increment_fail()` devuelve `0`. El login no se interrumpe por una falla de Redis.

#### Archivos creados / modificados — Backend

**`backend/app/core/config.py`** — nuevas variables:

```python
LOGIN_MAX_ATTEMPTS: int = 5
LOGIN_BLOCK_MINUTES: int = 5
```

**`backend/app/core/messages.py`** — nuevo mensaje:

```python
ACCOUNT_LOCKED = "Cuenta bloqueada temporalmente por demasiados intentos fallidos. Inténtalo de nuevo en {minutes} minutos."
```

**`backend/app/core/login_limiter.py`** — módulo nuevo:

Tres funciones públicas:

| Función | Descripción |
|---|---|
| `is_blocked(email)` | Comprueba si existe la clave de bloqueo en Redis |
| `increment_fail(email)` | Incrementa el contador; crea la clave de bloqueo si llega al límite |
| `reset_fails(email)` | Elimina ambas claves tras login exitoso |

**`backend/app/api/routes/login.py`** — lógica de login reescrita:

```python
@router.post("/login/access-token")
@limiter.limit("5/minute")
def login_access_token(..., remember_me: bool = Query(False)) -> Token:
    email = form_data.username

    if login_limiter.is_blocked(email):          # 429 si bloqueado
        raise HTTPException(429, ...)

    user = crud_user.authenticate(...)
    if not user:
        count = login_limiter.increment_fail(email)
        remaining = max(0, LOGIN_MAX_ATTEMPTS - count)
        if remaining == 0:
            raise HTTPException(429, ...)        # bloqueo al 5° fallo
        raise HTTPException(400, f"... Intentos restantes: {remaining}")
    elif not user.is_active:
        raise HTTPException(403, INACTIVE_USER)

    login_limiter.reset_fails(email)             # reset tras éxito
    ...
    _set_refresh_cookie(response, refresh_token, persistent=remember_me)
```

#### Archivos modificados — Frontend

**`frontend/src/app/features/auth/login/login.ts`**

- `blocked = signal(false)` + `blockMessage = signal('')`
- En el handler de error: si el detalle contiene "bloqueada" (o status 429), activa `blocked` en lugar de mostrar toast
- La detección usa contenido del mensaje porque el status HTTP se pierde al pasar por el NgRx action system

**`frontend/src/app/features/auth/login/login.html`**

Alerta roja con ícono de candado, visible solo cuando `blocked()` es `true`:


### 3.3 Auto-redirección al Dashboard con Sesión Activa

#### Problema

Cuando el usuario tenía "Recordarme" activado, el cookie persistente permitía renovar la sesión al abrir el navegador. Sin embargo, al navegar a `/auth/login`, el formulario se mostraba igualmente y el usuario tenía que hacer clic en "Iniciar sesión" para continuar.

#### Solución

En `LoginComponent.ngOnInit()` se añadieron dos comprobaciones:

```typescript
ngOnInit(): void {
  // 1. Token ya en memoria (misma sesión, recarga de página)
  if (getAccessToken()) {
    void this.router.navigateByUrl(this.auth.getDefaultRedirectPath());
    return;
  }

  // 2. Refresh en curso (cookie persistente, navegador recién abierto)
  this.store.select(selectAuthInitialized).pipe(
    filter((init) => init),
    take(1),
  ).subscribe(() => {
    if (getAccessToken()) {
      void this.router.navigateByUrl(this.auth.getDefaultRedirectPath());
    }
  });
}
```

**Flujo completo con "Recordarme":**

```
Navegador abre → App carga → App.constructor dispara refreshToken
                                         ↓
                              Effect llama POST /api/v1/auth/refresh
                              (con cookie persistente httpOnly)
                                         ↓
                              Backend valida → nuevo access token
                              → loadCurrentUser → initialized=true
                                         ↓
LoginComponent.ngOnInit() → selectAuthInitialized emite true
→ getAccessToken() !== null → navigateByUrl('/dashboard')
```

Sin "Recordarme" (o cookie expirado): el refresh falla → `initialized=true` pero sin token → formulario se muestra normalmente.

18. Para desbloquear manualmente en Redis: `DEL circula:local:login_block:{email}`
